# Stage 1：解析 NotebookLM 報告

## 你的任務

讀取使用者貼入的 NotebookLM 研究報告（散文格式），拆解並重組為結構化事實清單。

**不要被 NotebookLM 的語氣或框架帶著走。**
你的目標是「提取事實」，而不是「摘要它的摘要」。

---

## 提取步驟

閱讀整份報告後，輸出以下格式：

```
## 事件解析

**核心事件（一句話）**：[誰、做了什麼、何時何地]

---

**背景脈絡**
- [相關前因、歷史背景、這件事發生前的狀態]
- [涉及的關鍵組織、人物、政策]

**事件經過**
- [時間線或關鍵節點]
- [已確認的行動或決定]

**數據與具體細節**
- [數字、金額、規模、日期等可查證的事實]

**影響與後果**
- [已發生的直接影響]
- [預期中的後續發展]
```

---

## 兩份清單（重要）

事實清單輸出後，接著列出兩份清單。

---

### 清單一：讀者可能不知道的切入角度與背景認知

這些不是名詞解釋表，而是「理解這件事所需要的認知前提」。
目標是找出：報告裡有哪些東西，是建立在讀者可能不具備的背景知識之上的？

掃描方式：逐段問自己「這個說法成立，需要讀者先知道什麼？」

```
## 清單一：讀者可能不知道的切入角度與背景認知
（決定摘要裡哪些前提要交代、哪些可以略過）

□ [描述這個認知缺口]：[為什麼不知道這個，這件事就看不懂？]
□ [描述這個認知缺口]：[同上]
□ [描述這個認知缺口]：[同上]
```

範例風格（不是名詞解釋，而是認知框架）：
- 錯：「BIS：國際清算銀行，是各國央行的央行」
- 對：「大多數人以為各國央行各自決策，但這件事要成立，需要知道 BIS 長期扮演跨國政策協調的角色」

---

### 清單二：從報告中提煉的有趣知識點

不是泛泛的「接下來會怎樣」，而是從**這份具體報告**裡找出：
讀者讀到這裡，哪些細節或角度會讓他自然想往下挖？

掃描方式：找報告裡那些「說了一半」或「一帶而過但其實很關鍵」的東西。

```
## 清單二：從報告中提煉的有趣知識點
（決定摘要的「為什麼」要從哪個角度切入）

□ [具體的知識點或細節]：[它為什麼有趣？報告裡有沒有足夠的資訊支撐？]
□ [具體的知識點或細節]：[同上]
□ [具體的知識點或細節]：[同上]
```

範例風格（來自報告本身，不是泛泛提問）：
- 錯：「接下來這個政策會怎麼走？」
- 對：「報告提到這個決定繞過了原本的審議程序——這個細節說明了為什麼反對聲音這麼快就被壓下去」

---

## 輸出後詢問

「以上是事實清單、知識背景清單、好奇心清單。

- 清單一：告訴我哪些你已經知道，哪些需要我補充——這決定摘要裡要交代多少背景。
- 清單二：告訴我你最想在摘要裡看到哪個問題的答案——這決定『為什麼』那個部分的切入角度。

或者直接輸入 `/draft`，我會根據現有資訊判斷。」
