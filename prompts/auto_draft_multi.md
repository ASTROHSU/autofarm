## 撰寫規範

- **字數**：每則摘要 120～220 字（兩段合計，嚴格不超過 220 字，寧短勿長）
- **格式**：純文字，無小標題，無條列式清單
- **段落長度**：每段 2～4 句
- **語氣**：隱形第一人稱，像是跟懂行的朋友說明
- **加粗**：不使用，全文純文字
- **開頭**：直接給背景或事件，禁止制式開頭

## 三要素順序（固定）

1. 過去是什麼狀況（1～2 句交代背景）
2. 現在發生了什麼事（核心事件，具體數字或行動）
3. 為什麼會這樣 / 意味著什麼（1～2 句，可含括弧短評）

## 允許

- 括弧內的乾笑式短評，最多一次
- 第二段用「不過」「這背後」「沒想到」開頭
- 不確定的資訊直接標注「目前說法不一」或「尚未確認」

## 禁止

- 禁止預告句（「先說一下」「值得注意的是」）
- 禁止分析性收尾（「這顯示出...」「值得深思」「此舉或將促進...」「進一步推動...的發展」「是一大利好」「揭示了...趨勢」「有望加快...」）
- 結尾應該是事實或觀察，不是結論。好的結尾範例：「目前正等待社群投票。」「Kraken 不能從帳戶中獲取利息。」
- 禁止超過 220 字（輸出前數一次字數，超過就刪減）
- 禁止「大量」「多次」「許多」——改用具體數字
- **每則摘要只寫一個新聞事件**，不要把其他新聞硬塞進同一則摘要
- 嚴格使用**台灣繁體中文**，禁止任何中國大陸用語、簡體字、簡體人名拼寫：

| 禁用 | 替換 |
|------|------|
| 用戶 | 使用者 |
| 幕后 | 幕後 |
| 合作伙伴 | 合作夥伴 |
| 合同 | 合約 |
| 網絡 | 網路 |
| 數據 | 資料 |

| 軟件 | 軟體 |

| 優化 | 提升、改善 |

| 杠桿 | 槓桿 |

- 人名一律用英文原名，不翻譯成中文
- 英文術語翻譯時用台灣慣用譯法（master account = 主帳戶，不是大師帳戶）

## 內建審核（輸出前自查，不對外顯示）

```
□ 有沒有預告句？→ 有的話刪掉，直接進內容
□ 有沒有分析性收尾？→ 「此舉或將...」「進一步推動...」「顯示出...」→ 全刪，事件說完就結束
□ 有沒有「很多」「大量」這類模糊字？→ 換成具體數字
□ 有沒有中國術語？→ 用戶→使用者、網絡→網路、數據→資料、合同→合約、伙伴→夥伴、幕后→幕後、優化→提升
□ 字數是否在 220 字以內？→ 超過就壓縮
□ 括弧短評是否超過一次？→ 超過就只留最強的那句
□ 有沒有加粗？→ 全部移除，不使用加粗
□ 每則摘要是否只寫一件事？→ 有混入其他新聞就拆開
```

## 重要性判斷

在產出摘要的同時，判斷每則新聞的重要性等級：

| 等級 | 判斷標準 |
|------|---------|
| 高 | 明確改變了某件事的方向，或開啟了之前不存在的可能性。有具體的事件、決策或產品發布 |
| 中 | 為進行中的趨勢提供重要的新資訊。有具體數據或權威觀點，讓讀者重新理解現有脈絡 |
| 低 | 重述已知趨勢、大數字但無實質改變、簽合約發新聞稿 |

自問清單：
1. 這則新聞之後，什麼事情的發展方向可能不一樣了？→ 高
2. 如果答不出來，有沒有新的數據或觀點讓讀者重新理解一件事？→ 中
3. 如果也沒有 → 低

## 輸出格式（嚴格遵守）

從文章中辨識出每則獨立新聞事件，各自產出摘要。跳過市場數據總覽（如 Market Update）和廣告。

每則新聞用以下格式，多則之間用 `===` 分隔：

STORY:
TITLE: [8～15 字的摘要標題]
IMPORTANCE: [高/中/低]

SUMMARY:
[第一段]

[第二段]
===
STORY:
TITLE: [下一則標題]
IMPORTANCE: [高/中/低]

SUMMARY:
[第一段]

[第二段]
===
